# Demo Site

`demo_site/` — веб-консоль, которая наглядно показывает, как сообщения ходят
по SITL-пайплайну (`verifier → controller → redis → core → messaging`) и как
ядро симулирует движение роя дронов.

В отличие от «закрытой» анимации здесь каждая метрика соответствует конкретным
топикам брокера: операторы и демо-гости видят реальный объём команд, ответы
ядра, отклонения верификатора и задержку отклика.

---

## 1. Запуск

### MQTT (по умолчанию)

```bash
make up-mqtt      # поднимет mosquitto
make up-sitl      # redis + verifier + controller + core + messaging
make up-demo      # demo_site на http://localhost:8080
```

### Kafka

```bash
BROKER_BACKEND=kafka make up-kafka
BROKER_BACKEND=kafka make up-sitl
BROKER_BACKEND=kafka make up-demo
```

### Остановить и всё почистить

```bash
make down           # остановить контейнеры
make clean          # снести контейнеры и тома (включая pip_cache)
```

### Перезапуск только демо-сервиса

После правок кода в `demo_site/` достаточно:

```bash
docker compose restart demo_site
```

Если поменялись зависимости — `docker compose up -d --build demo_site`.
Named-volume `pip_cache` ускоряет последующие пересборки.

## 2. Что показывает интерфейс

### Верхняя панель

- **Сценарий** — выпадашка: Хоровод / Восьмёрка / V-строй / Разлёт / Сбор.
  Переключение применяется в реальном времени, без остановки роя.
- **Дронов** — сколько дронов поднять при нажатии «Запустить» (1–16).
- **Запустить / Остановить** — жизненный цикл авто-сценария.
- **broker на связи / broker недоступен** — состояние подключения к шине.
- Счётчики потока: `→ команд` (отправлено в `sitl.commands`),
  `← ответов` (получено из `sitl.telemetry.response.demo`),
  `msg/s` — скользящая оценка трафика за последние 5 секунд.

### Полоса пайплайна

Пять узлов подсвечиваются зелёным, когда через них недавно
(≈2 с) проходил трафик, и становятся тусклыми, если пайплайн замер:

| Узел       | Чем активируется                                 |
|------------|--------------------------------------------------|
| Runner     | публикация `COMMAND` из авто-сценария            |
| Verifier   | приход `sitl.verified-commands` / `verified-home`|
| Controller | движение любого дрона за окно 2 с                |
| Core       | то же, что Controller (смена позиций)            |
| Messaging  | недавний ответ на `POSITION_REQUEST`             |

Число у каждого узла — накопительный счётчик с момента последнего старта демо.

### Карта

Тёмное поле с сеткой 60 px. На ней:

- крестики `+` — HOME-точки дронов;
- цветные точки — текущее положение + стрелка направления;
- полупрозрачный след длиной до 60 точек;
- подпись `D01`, `D02`, … для авто-дронов и `M01`, `M02`, … для «ручных»;
- при отсутствии брокера показывается соответствующее сообщение,
  при нулевом количестве дронов — «Нажмите Запустить…».

### Правая панель

- список дронов с координатами, скоростью, высотой и курсом;
  напротив ручных дронов — жёлтый бейдж **manual**;
- вкладка «Ручная команда» (подробнее — раздел 4).

## 3. Сценарии

Каждый сценарий — это чистая функция `(layout, t, total) → (vx, vy, vz)`
в `demo_site/scenarios.py`. Раннер дёргает её раз в `command_interval_sec`
(0.75 с) для всех auto-дронов и публикует `COMMAND` в `sitl.commands`.

| ID          | Описание                                                       |
|-------------|----------------------------------------------------------------|
| `orbit`     | «Хоровод»: синхронная орбита роя с радиальной пульсацией.      |
| `figure8`   | «Восьмёрка»: каждый дрон трассирует лемнискату со сдвигом фаз. |
| `formation` | «V-строй»: коллективный марш с покачиванием курса ±70°.        |
| `spread`    | «Разлёт»: синусоидальное радиальное движение от центра.        |
| `gather`    | «Сбор»: инверсия разлёта — стартуем к центру.                  |

Границы по скорости (`±50 м/с` по оси, `±10 м/с` по `vz`) навязаны
JSON-схемой `sitl-commands.json` — `build_command_payload` делает
`_clamp`, чтобы ни один сценарий не упирался в отклонение верификатором.

Добавить новый сценарий — объявить функцию-обработчик, положить в словарь
`SCENARIOS` и добавить пункт в `SCENARIO_LIST`. UI подтянет его
автоматически через `GET /api/state`.

## 4. Ручное управление

Панель «Ручная команда» (раскрывается кликом по заголовку) имеет три
вкладки:

### 4.1. «Скорость»

Отправляет одиночный `COMMAND` выбранному дрону, минуя авто-сценарий:

- `vx`, `vy`, `vz` — компоненты скорости (м/с);
- `курс°` — магнитный курс; если поле пустое, курс рассчитывается из `vx/vy`.

Авто-сценарий на следующем такте «перебьёт» команду, если дрон остался
auto. Чтобы «зависнуть» на ручной траектории, сценарий надо поставить на
паузу («Остановить»).

### 4.2. «HOME»

Публикует HOME выбранному дрону:

- `lat`, `lon`, `alt` — базовая точка.

После верификации HOME приходит в `sitl.verified-home`, где его ловит
сам демо-сервис — см. раздел 5.

### 4.3. «Сырое»

Нижнеуровневая публикация произвольного JSON в один из трёх топиков:
`sitl.commands`, `sitl-drone-home`, `sitl.telemetry.request`.

Сервер делает локальную проверку `shared.contracts.validate_schema`
(той же функцией, что и верификатор). В статусе формы отображается:

- `✓ схема <имя>: OK → верификатор примет` — если валиден;
- `✗ отправлено, но верификатор отклонит: <reason>` — если нет.

Сообщение в любом случае уходит в брокер — это именно «сырая» публикация
для демонстрации поведения верификатора.

Готовые пресеты (8 штук): валидные команда/HOME/запрос и заведомо
«битые» (пропущенные поля, выход за диапазон, мусор, неверный формат
`drone_id`). Пресет заполняет topic + payload, текст можно править.

## 5. Авто-discovery дронов через ручной HOME

Сценарий поднимает фиксированные ID (`drone_901 … drone_916`). Ручной
HOME может завести новый ID, и он автоматически появится на карте.

**Как это работает.** Demo-сервис подписан на `sitl.verified-home`
(`Simulation._on_verified_home` → `_record_verified_home`). Когда
верификатор пропускает HOME, demo:

1. Если `drone_id` уже в `_layouts` — обновляет координаты HOME.
2. Если новый — создаёт `DroneLayout(auto=False)` с меткой `M01…`
   и собственным цветом. Дрон тут же попадает в `_poll_loop`,
   тот начинает опрашивать его позицию через `sitl.telemetry.request`.

Авто-сценарий такие дроны не трогает (`auto=False` исключён из
`_command_loop`), и `start()` их не теряет: повторный запуск сценария
сохраняет «ручных» наравне с новыми auto-дронами.

**Как показать в демо:**

1. Не нажимая «Запустить», открыть панель → вкладку «Сырое».
2. Выбрать пресет «✓ Валидный HOME», поправить `drone_id` (напр. `drone_777`).
3. «Отправить в брокер». Через ~0.4–1 с в списке дронов появляется
   `M01 · drone_777 [manual]`, на карте — HOME-крест и точка.
4. Во вкладке «Скорость» выбрать этого дрона и подать ненулевую скорость —
   он поедет отдельно от роя.
5. Нажать «Запустить» — ручные дроны сохраняются, рядом запускается auto-рой.

**Демонстрация отклонения.** Если в «Сыром» отправить пресет
«✗ HOME без home_alt», верификатор отклонит — счётчик `Verifier` не
увеличится, новый дрон на карте не появится; статус формы подсветит
причину локальной валидации.

## 6. HTTP API

Все ручки возвращают текущий snapshot (как `GET /api/state`) — удобно
для автотестов.

| Метод | Путь                   | Назначение                                              |
|-------|------------------------|---------------------------------------------------------|
| GET   | `/api/state`           | Полный снимок: дроны, метрики, сценарий, список сценариев |
| POST  | `/api/start`           | Поднять авто-рой. Тело: `{"count": 6}`                  |
| POST  | `/api/stop`            | Остановить авто-сценарий; ручных дронов не трогает      |
| POST  | `/api/scenario`        | Сменить сценарий. Тело: `{"id": "figure8"}`             |
| POST  | `/api/manual/command`  | Одиночная скорость. `{drone_id, vx, vy, vz?, mag_heading?}` |
| POST  | `/api/manual/home`     | Одиночный HOME. `{drone_id, home_lat, home_lon, home_alt}` |
| POST  | `/api/raw/publish`     | Сырая публикация. `{topic, payload}`                    |

Топики в `/api/raw/publish`: `sitl.commands`, `sitl-drone-home`,
`sitl.telemetry.request`. Ответ включает `local_valid` (bool) и `reason`
(string) от `shared.contracts.validate_schema`.

## 8. Частые проблемы

| Симптом                                    | Куда смотреть                                          |
|--------------------------------------------|--------------------------------------------------------|
| `broker недоступен`                        | `docker compose ps`; перезапустить `mosquitto` / `kafka` |
| «→ команд» растёт, `← ответов` — нет       | `docker logs sitl-messaging` — должен отвечать на `request_position` |
| Дроны появляются, но не двигаются          | `docker logs sitl-core` — проверить интеграцию 10 Гц   |
| Верификатор молча не пропускает            | `docker logs sitl-verifier` — видно причину отклонения |
| Raw HOME не создаёт дрона                  | Счётчик Verifier должен инкрементнуться; если нет — смотри логи verifier |
| Правки `demo_site/*.py` не подтянулись     | `docker compose restart demo_site` (или `--build` при смене зависимостей) |
| Статика не обновилась                      | Ctrl+F5 в браузере — index.html кэшируется             |
