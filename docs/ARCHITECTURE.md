# Архитектура New-SITL (Pure)

## sitl_verifier

**Ответственность:** Приём raw-сообщений из брокера, валидация по JSON-схемам, публикация verified-сообщений.

**Цели безопасности (ЦБ):**
1. Все входящие сообщения проходят JSON Schema валидацию по схемам из `schemas/`
2. Только verifier публикует в verified-топики (`sitl.verified-commands`, `sitl.verified-home`)
3. Rejected-сообщения логируются для аудита

**Предположения безопасности (ПБ):**
1. JSON-схемы покрывают все обязательные и опциональные поля
2. Raw-топики доступны только для чтения verifier
3. Verified-топики доступны только для записи verifier

---

## sitl_controller

**Ответственность:** Приём verified HOME/COMMAND сообщений, сохранение состояния дронов в Redis.

**Цели безопасности (ЦБ):**
1. Запись только в `drone:{id}:state` с валидацией структуры
2. COMMAND без предварительного HOME игнорируется
3. Controller читает только verified-топики

**Предположения безопасности (ПБ):**
1. Redis ACL ограничивает controller только SET/HSET/HGETALL для `drone:*`
2. Verified-топики гарантируют прохождение через verifier
3. TTL ключей устанавливается при создании и продлевается при обновлениях

---

## sitl_core

**Ответственность:** Периодическое обновление позиций движущихся дронов (10Hz) в Redis.

**Цели безопасности (ЦБ):**
1. SCAN с COUNT параметром для предотвращения блокировки Redis
2. Обновление только дронов со статусом `MOVING`
3. Векторы `vx`, `vy`, `vz` читаются из валидированного состояния

**Предположения безопасности (ПБ):**
1. Redis доступен из Docker network без внешнего доступа
2. SCAN используется с `COUNT` для предотвращения DoS
3. Ключи `drone:*:state` имеют корректный TTL, stale-состояния очищаются

---

## sitl_messaging

**Ответственность:** Обработка запросов телеметрии, ответ текущей позицией в `reply_to`.

**Цели безопасности (ЦБ):**
1. Ответ отправляется только в указанный `reply_to` topic
2. `correlation_id` сохраняется в ответе (payload + headers для Kafka)
3. Запросы для несуществующих дронов логируются и отклоняются

**Предположения безопасности (ПБ):**
1. Redis ACL разрешает messaging только READ операции для `drone:*`
2. `reply_to` topic указывается отправителем запроса
3. Correlation_id уникален для каждого запроса

---

## SystemBus (Kafka/MQTT)

**Ответственность:** Единая шина коммуникации между всеми компонентами.

**Цели безопасности (ЦБ):**
1. SASL/SSL для Kafka, username/password для MQTT
2. Разные credentials для разных компонентов (read/write per topic)
3. Timestamp и nonce в сообщениях, проверка уникальности correlation_id

**Предположения безопасности (ПБ):**
1. Брокер доступен только из Docker network
2. Consumer groups уникальны для каждого окружения
3. QoS=1 для MQTT критичных сообщений
4. Kafka topic ACL настроены для каждого компонента

---

## Redis

**Ответственность:** Хранилище текущего состояния всех дронов.

**Цели безопасности (ЦБ):**
1. Только controller записывает, core обновляет, messaging читает
2. SCAN с лимитом, TTL для автоматической очистки stale
3. Нет FLUSHALL/KEYS для приложений, только SET/GET/HGETALL

**Предположения безопасности (ПБ):**
1. Redis ACL настроен для каждого компонента с минимальными привилегиями
2. Ключи следуют паттерну `drone:{id}:state` с фиксированной структурой
3. TTL=2ч по умолчанию, продлевается при каждом обновлении
