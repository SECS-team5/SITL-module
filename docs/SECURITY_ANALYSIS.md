# Анализ безопасности системы New-SITL

## sitl_verifier

**Ответственность:** Валидация входящих raw-сообщений по JSON-схемам, публикация verified-сообщений.

**Цели безопасности (ЦБ):**
1. Целостность JSON Schema валидации — все входящие сообщения проходят проверку по схемам из `schemas/`
2. Аутентичность verified-сообщений — только verifier публикует в `sitl.verified-*` топики
3. Логирование rejected-сообщений — все отклонённые сообщения фиксируются для аудита

**Предположения безопасности (ПБ):**
1. JSON-схемы в `schemas/` покрывают все обязательные и опциональные поля
2. Raw-топики (`sitl.commands`, `sitl-drone-home`) доступны только для чтения verifier
3. Verified-топики (`sitl.verified-*`) доступны только для записи verifier

---

## sitl_controller

**Ответственность:** Приём verified HOME/COMMAND сообщений, сохранение состояния дронов в Redis.

**Цели безопасности (ЦБ):**
1. Целостность Redis ключей — запись только в `drone:{id}:state` с валидацией структуры
2. Авторизация HOME/COMMAND — COMMAND без предварительного HOME игнорируется
3. Изоляция от raw-топиков — controller читает только verified-топики

**Предположения безопасности (ПБ):**
1. Redis ACL ограничивает controller только операциями SET/HSET/HGETALL для `drone:*`
2. Verified-топики гарантируют прохождение через verifier (метка доверия)
3. TTL ключей устанавливается при создании и продлевается при обновлениях

---

## sitl_core

**Ответственность:** Периодическое обновление позиций движущихся дронов (10Hz) в Redis.

**Цели безопасности (ЦБ):**
1. Rate limiting SCAN — операция SCAN не блокирует Redis (COUNT параметр, лимит 90мс)
2. Ограничение доступа — core обновляет только дроны со статусом `MOVING`
3. Целостность расчёта позиций — векторы `vx`, `vy`, `vz` читаются из валидированного состояния

**Предположения безопасности (ПБ):**
1. Redis доступен из Docker network без внешнего доступа
2. SCAN используется с `COUNT` для предотвращения DoS
3. Ключи `drone:*:state` имеют корректный TTL, stale-состояния очищаются

---

## sitl_messaging

**Ответственность:** Обработка запросов телеметрии, ответ текущей позицией в `reply_to`.

**Цели безопасности (ЦБ):**
1. Конфиденциальность reply_to — ответ отправляется только в указанный topic
2. Целостность correlation_id — идентификатор запроса сохраняется в ответе (payload + headers для Kafka)
3. Защита от unknown drone — запросы для несуществующих дронов логируются и отклоняются

**Предположения безопасности (ПБ):**
1. Redis ACL разрешает messaging только READ операции для `drone:*`
2. `reply_to` topic указывается отправителем запроса (доверенный источник)
3. Correlation_id уникален для каждого запроса (защита от replay)

---

## SystemBus (Kafka/MQTT)

**Ответственность:** Единая шина коммуникации между всеми компонентами.

**Цели безопасности (ЦБ):**
1. Конфиденциальность транспорта — SASL/SSL для Kafka, username/password для MQTT
2. Topic isolation — разные credentials для разных компонентов (read/write per topic)
3. Защита от replay атак — timestamp и nonce в сообщениях, проверка уникальности correlation_id

**Предположения безопасности (ПБ):**
1. Брокер доступен только из Docker network
2. Consumer groups уникальны для каждого окружения (например, `sitl-verifier-v1`)
3. QoS=1 для MQTT критичных сообщений (гарантированная доставка)
4. Kafka topic ACL настроены для каждого компонента

---

## Redis

**Ответственность:** Хранилище текущего состояния всех дронов.

**Цели безопасности (ЦБ):**
1. Целостность данных — только controller записывает, core обновляет, messaging читает
2. Доступность — SCAN с лимитом, TTL для автоматической очистки stale
3. Запрет опасных операций — нет FLUSHALL/KEYS для приложений, только SET/GET/HGETALL

**Предположения безопасности (ПБ):**
1. Redis ACL настроен для каждого компонента (controller: write, core: scan+update, messaging: read)
2. Ключи следуют паттерну `drone:{id}:state` с фиксированной структурой
3. TTL=2ч по умолчанию, продлевается при каждом обновлении

## Рекомендации

1. **Включить SASL_SSL** для Kafka в production
2. **Redis ACL** для каждого компонента с минимальными привилегиями
3. **Rate limiting** на SCAN и запись в Redis
4. **Correlation ID + timestamp** во всех сообщениях для защиты от replay
5. **Мониторинг** rejected сообщений и аномалий в Redis
