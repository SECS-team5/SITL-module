# 3. Описание внедрённых шаблонов СКИБ

## Обзор

В системе SITL применены следующие шаблоны проектирования СКИБ (систем с конструктивной информационной безопасностью) в соответствии с требованиями ГОСТ Р по обеспечению безопасности на уровне архитектуры.

---

## Шаблон 1: Chain of Responsibility (Цепочка ответственности)

**Компонент:** sitl_verifier

**Назначение:** Последовательная проверка входных сообщений через цепочку обработчиков

**Реализация в системе:**
- Парсинг JSON → Классификация по топику → Валидация по схеме → Публикация результата

**Этапы цепочки:**

- **Step 1: JSON Parse** — Валидация: структура JSON. Результат: словарь или отклонение
- **Step 2: Topic Classification** — Валидация: определение типа (command/home). Результат: message_type + schema_name или отклонение
- **Step 3: Schema Validation** — Валидация: JSON Schema Draft 2020-12. Результат: валидный payload или отклонение
- **Step 4: Publish Verified Message** — Действие: публикация в verified-топик. Результат: верифицированное сообщение

**Поток данных:** Raw Message → JSON Parse → Topic Classification → Schema Validation → Publish Verified Message → Verified Message

**Влияние на ДВБ:** Средняя — каждый этап цепочки добавляет уровень защиты, отклонение на любом этапе предотвращает проникновение невалидных данных

**Критичность:** Высокая — единственная точка входа для всех входных данных

---

## Шаблон 2: Memento (Снимок состояния)

**Компонент:** sitl_controller, sitl_core

**Назначение:** Криптографически стойкое хранение и восстановление состояния home-позиции дрона

**Реализация в системе:**
- При установке HOME: сохранение полного состояния (home_lat, home_lon, home_alt, mag_heading)
- При перезапуске: восстановление состояния из Redis
- При откате: использование сохранённого home для определения границ геозоны

**Роли:**
- **Originator:** sitl_core — создаёт и восстанавливает снимок
- **Memento:** home_state — неизменяемый объект состояния
- **Caretaker:** Redis — хранение снимка

**Содержимое Memento:**
```json
{
"status": "ARMED",
"home_lat": 55.7558,
"home_lon": 37.6173,
"home_alt": 100.0,
"mag_heading": 45.0,
"last_update": "2026-01-15T10:30:00Z"
}
```

**Операции Memento:**
1. **Save:** При получении HOME — сохранение начального состояния дрона
2. **Restore:** При обработке COMMAND — проверка что home существует
3. **Boundary Check:** При обновлении позиции в CORE — проверка нахождения в пределах геозоны

**Влияние на ДВБ:** Средняя — обеспечивает целостность home-позиции и возможность аудита изменений

**Критичность:** Средняя — защищает от подмены home-позиции

## Шаблон 3: Strategy (Стратегия)

**Компонент:** sitl_controller

**Назначение:** Подмена алгоритмов принятия решений на лету в зависимости от состояния дрона

**Реализация в системе:**
- Стратегия ARMED: дрон принимает команды движения
- Стратегия MOVING: дрон игнорирует новые команды до остановки
- Стратегия HOME_MISSING: COMMAND отклоняются без HOME

**Стратегии:**
- **ARMED:** Принять команду → MOVING
- **MOVING:** Игнорировать новые команды
- **HOME_MISSING:** Отклонить COMMAND без HOME

**Контекст:** sitl_controller

**Правила переходов:**

| Текущий статус | Входная команда | Следующий статус | Действие |
|---------------|-----------------|------------------|----------|
| ARMED | Не нулевая скорость | MOVING | Применить команду |
| ARMED | Нулевая скорость | ARMED | Обновить heading |
| MOVING | Любая | MOVING | Игнорировать |
| любой | HOME | ARMED | Сбросить позицию |

**Влияние на ДВБ:** Высокая — определяет поведение дрона в зависимости от контекста

**Критичность:** Высокая — непосредственно влияет на физическое поведение дронов

## Шаблон 4: Proxy (Заместитель)

**Компонент:** sitl_messaging

**Назначение:** Перехват запросов телеметрии для добавления контроля целостности и валидации

**Реализация в системе:**
- Валидация входящего запроса по схеме
- Проверка наличия состояния дрона
- Сохранение correlation_id для traceability
- Контроль ответного топика (reply_to)

**Схема взаимодействия:** Client → TelemetryProxy → RealSub с параллельными операциями Validate Request и Log Request

**Функции Proxy:**
1. Валидация структуры запроса
2. Проверка полномочий (наличие drone_id)
3. Tracing (correlation_id propagation)
4. Контроль места отправки ответа

**Влияние на ДВБ:** Высокая — контролирует доступ к данным о позиции дронов

**Критичность:** Высокая — защищает от несанкционированного доступа к телеметрии

---

## Шаблон 5: Broker (Посредник)

**Компонент:** SystemBus (Kafka/MQTT)

**Назначение:** Единая шина коммуникации, разделяющая команды, телеметрию и события

**Реализация в системе:**
- Топик-маршрутизация: raw → verified → consumer
- Транспортная абстракция: одинаковый API для Kafka и MQTT
- Изоляция компонентов через consumer groups

**Поток данных:** Producer (Drone Pilots) → SystemBus (Kafka/MQTT) → распределение по топикам → Verifier / Controller / Messaging

**Топиковая модель:**

| Топик | Назначение | Потребители | Тип |
|-------|-----------|-------------|-----|
| sitl.commands | Входные команды дронов | verifier | raw |
| sitl-drone-home | Домашние позиции | verifier | raw |
| sitl.verified-commands | Верифицированные команды | controller | trusted |
| sitl.verified-home | Верифицированные HOME | controller | trusted |
| sitl.telemetry.request | Запросы позиций | messaging | request |
| sitl.telemetry.response | Ответы позиций | external | response |

**Влияние на ДВБ:** Средняя — обеспечивает изоляцию потоков данных

**Критичность:** Средняя — сбой брокера приводит к потере коммуникации

## Соответствие шаблонов принципам КИБ

**Принцип 1: Безопасность зарождается с момента появления идеи**

Все шаблоны интегрированы в архитектуру на этапе проектирования:
- Chain of Responsibility предотвращает попадание невалидных данных
- Broker изолирует компоненты друг от друга

**Принцип 2: Минимизация поверхности атаки**

- Memento ограничивает доступ к состоянию только через определённые операции
- Proxy перехватывает все запросы телеметрии

**Принцип 3: Защита на уровне конструктивных решений**

- Strategy делает невозможным некорректное управление дроном
- Broker предотвращает cross-contamination топиков

---

## Рекомендации по усилению шаблонов

1. **Chain of Responsibility:** Добавить HMAC-подпись верифицированных сообщений
2. **Memento:** Хранить историю изменений home-позиции с timestamping
3. **Strategy:** Добавить валидацию геозоны при каждом COMMAND
4. **Proxy:** Внедрить rate limiting для запросов телеметрии
5. **Broker:** Настроить ACL для каждого топика по принципу минимальных привилегий
